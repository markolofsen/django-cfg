# Server-Side Capture

The monitor backend captures server-side errors through multiple automatic hooks and a manual `capture` API. All paths are **fire-and-forget** — they never raise and never break the calling code.

---

## Automatic Capture Paths

### HTTP 500 Errors

Unhandled exceptions that produce a 500 response are captured via Django's `got_request_exception` signal. Request metadata (URL, method, status code) is attached automatically.

### Logging Handler

The `MonitorHandler` integrates with Python's standard `logging` module. It captures:

- `ERROR` records without an exception → stored as `LOG_ERROR` events
- `ERROR` / `CRITICAL` records with an attached exception → stored as `UNHANDLED_EXCEPTION` events

A thread-local reentrancy guard prevents infinite recursion if the capture itself logs.

### Slow Queries

Database queries exceeding the configured threshold are captured as `SLOW_QUERY` events. The wrapper is registered on every new database connection via Django's `connection_created` signal.

Default threshold: **500 ms**. Set to `0` to disable.

### RQ Job Failures

Failed background jobs are captured as `RQ_FAILURE` events. The handler is registered as a global RQ exception handler — no per-queue configuration needed.

### OOM / SIGKILL

`MonitoringWorker` overrides `work_horse_killed_handler`. When an RQ worker's child process is killed (OOM or SIGKILL), a `OOM_KILL` event is created.

To use:

```python
from django_cfg.apps.system.monitor.capture.rq import get_monitoring_worker_class

Worker = get_monitoring_worker_class()
Worker.work(queues=["default"])
```

## Manual Capture

### Exceptions

```python
from django_cfg.apps.system.monitor.capture import capture

try:
    process_payment(order)
except Exception:
    capture.exception(extra={"order_id": order.id})
```

Calling `capture.exception()` inside an `except` block automatically includes the current traceback.

### Messages

```python
capture.message(
    "Suspicious login attempt",
    level="warning",
    extra={"ip": ip, "user_id": user.pk},
)
```

## ServerEvent Model

Server-side errors are **deduplicated** — repeated occurrences of the same bug update a single row rather than creating new ones.

**Deduplication key:**

```
fingerprint = sha256("{exception_type}::{module}::{func_name}")[:16]
```

Line numbers are excluded from the fingerprint — the same bug deduplicates across refactors and deploys.

### Event Types

| Value | Capture mechanism |
|---|---|
| `SERVER_ERROR` | `got_request_exception` signal (HTTP 500) |
| `UNHANDLED_EXCEPTION` | `MonitorHandler` (logging with exception) |
| `SLOW_QUERY` | `execute_wrapper` |
| `RQ_FAILURE` | Global RQ `ExceptionHandler` |
| `OOM_KILL` | `MonitoringWorker.work_horse_killed_handler` |
| `LOG_ERROR` | `MonitorHandler` (non-exception ERROR logs) |

### Resolution and Regression Detection

Each `ServerEvent` has `is_resolved`, `resolved_at`, `resolved_by` fields. If a resolved event recurs, it is automatically **reopened** and its occurrence count incremented — no resolved errors are silently lost.

Bulk admin actions: **Mark resolved** / **Mark unresolved**.

## Separate Database Alias

<Callout type="warning">
Required when `ATOMIC_REQUESTS=True`. Under that setting, the main DB connection rolls back on HTTP 500, which would erase the `ServerEvent` write. Use a separate alias to keep monitor writes independent.
</Callout>

```python
# cfg.py
databases = {
    "default": DatabaseConfig.from_url(...),
    "monitor": DatabaseConfig.from_url(..., conn_max_age=0),  # same Postgres, no pooling
}
```

Configure the alias in monitor settings:

```python
frontend_monitor = FrontendMonitorConfig(
    monitor_db_alias="monitor",
)
```

## Safety Guarantees

| Concern | Solution |
|---|---|
| `AppRegistryNotReady` during startup | All model imports are lazy (inside functions) |
| Infinite recursion in `MonitorHandler` | Thread-local `in_emit` reentrancy guard |
| Slow query wrapper capturing itself | Thread-local `in_wrapper` reentrancy guard |
| ASGI / async context in `MonitorHandler` | `asyncio.get_running_loop()` + `sync_to_async` |
| Double registration on test suite reruns | `dispatch_uid` on all signals, attribute check on connections |
| `ATOMIC_REQUESTS` rollback on 500 | Separate `monitor_db_alias` connection |
| Broken ingest on notification failure | All `NotificationService` methods wrapped in `try/except` |
